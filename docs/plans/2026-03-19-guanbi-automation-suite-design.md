## 项目目标

构建一个本地运行的观远 BI 自动化套件。它不是单一脚本，而是一个可视化、配置驱动、阶段可开关、可持续扩展的本地工具，覆盖以下完整闭环：

1. 登录观远 BI
2. 拉取当前账号可访问的页面、目录、图表和筛选器定义
3. 通过本地界面选择需要导出的页面或图表
4. 生成并保存可复用模板
5. 按模板执行导出
6. 按需将导出结果写入本地 Excel 底板
7. 按需对 Excel 结果执行进一步处理
8. 按需将最终结果回写到飞书 Sheets

## 已确认结论

- 当前 `.env` 中的观远账号可以直接通过接口登录，不必把飞连扫码登录作为主实现路径。
- 观远当前站点存在稳定可调用的页面树、页面详情、导出任务和导出文件接口。
- 当前账号实测可访问 `312` 个页面，其中 `299` 个 `PAGE`，`13` 个 `CUSTOM_REPORT`。
- 普通页面和 `CUSTOM_REPORT` 页面都可以通过同一类页面详情接口读取。
- 页面详情中已经包含：
  - 页面级布局
  - tab / panel 关系
  - 图表定义
  - 筛选器定义
  - 筛选器到目标图表的字段映射
  - 数据源摘要
- 导出链路已实测可用，能够返回真实 `.xlsx` 二进制。
- 当前旧脚本可以证明“观远导出 -> Excel 底板 -> 飞书回写”业务链路成立，但实现高度写死、可维护性差。

## 当前核心问题

当前目录里的脚本存在以下系统性问题：

- 以单一业务为中心，配置和流程都写死在当前看板与当前底板上。
- 所有环节耦在一条脚本链路里，无法对单独阶段开关、重跑、替换。
- 环境声明和代码依赖不一致，项目不能稳定重建运行环境。
- Excel、飞书、观远三层逻辑交叉污染，无法独立验证。
- 当前没有统一的模板模型、运行归档模型、预检机制和错误边界。

详见 `docs/plans/2026-03-19-legacy-script-audit.md`。

## 方案比较

### 方案 1：继续扩展旧脚本

优点：

- 起步最快
- 短期最少改动

缺点：

- 很快会再次演变成更大的硬编码脚本
- 页面发现、筛选器建模、模板执行、Excel/飞书副作用无法解耦
- 后续新增阶段时会反复改主流程

结论：不推荐。

### 方案 2：本地 Web 工具 + 阶段化流水线

优点：

- 产品上是一体化工具，使用路径完整
- 工程上可分层、可插拔、可测试
- 支持阶段开关、重排、替换和未来插入新阶段
- 更适合长期维护和迭代

缺点：

- 第一版建设成本高于继续改脚本
- 需要先建立模板模型和运行模型

结论：推荐。

### 方案 3：先做“观远发现器”，后续再接执行

优点：

- 风险低
- 能快速确认元数据能力边界

缺点：

- 用户仍需在新旧工具间切换
- 最终还是要回到一体化

结论：可以作为子阶段，不作为最终产品形态。

## 当前推荐总体方案

采用“本地 Web 工具 + 阶段化流水线 + 模板驱动执行”的总体方案。

### 产品形态

- 本地运行
- 浏览器访问本地地址
- 不依赖云端中控
- 所有配置和运行归档保存在本地工作目录

### 工程原则

- 产品是一体的
- 架构是分层的
- 流水线是阶段化的
- 阶段是可开关的
- 阶段顺序应支持后续扩展
- 有副作用的阶段必须可以单独关闭

## 流水线设计

### 当前确认的标准阶段

第一版内建以下标准阶段：

1. `extract`
   - 登录观远
   - 获取页面/图表/筛选器定义
   - 渲染筛选条件
   - 执行导出
   - 归档下载文件

2. `workbook_ingest`
   - 将导出结果写入本地底板 Excel
   - 按模板中的写入规则覆盖、追加或定位写入

3. `workbook_transform`
   - 执行公式重算
   - 复制结果区域
   - 列裁剪、列映射、区域读取
   - 为发布阶段生成标准输出数据

4. `publish`
   - 将结果回写到飞书 Sheets
   - 记录批次上传结果

### 流水线不是固定三段式

系统不能把流程写死成 `1-2-3`。

正确的架构应允许未来变成：

- `1-2-3`
- `1-2-4-3`
- `1-4-2-3`
- `1-2`
- `1-3`

其中新增的 `4` 可以是未来补充的新标准阶段，例如：

- 文件规范化
- 中间数据清洗
- 自定义结果拆分
- 额外发布渠道

### 第一版的边界

第一版只支持“注册好的标准阶段”和它们的编排，不直接向用户开放“在界面里任意写 Python 脚本执行”。

原因：

- 任意代码执行的安全边界和调试成本过高
- 第一版更需要先把标准链路稳定下来
- 架构上要预留扩展口，但不必在第一版把“完全自定义执行器”做完

因此，第一版的灵活性目标是：

- 阶段可启用/禁用
- 阶段顺序可按规则编排
- 可新增新的内建阶段类型
- 任务模板可引用不同阶段组合

## 默认策略

不能把所有阶段做成全局默认开启，也不能全局默认关闭。

推荐策略如下：

### 新建模板

- 默认开启：`extract`
- 默认关闭：`workbook_ingest`
- 默认关闭：`workbook_transform`
- 默认关闭：`publish`

原因：

- 新模板刚创建时，下游目标通常还没配置完整
- Excel 和飞书阶段都有副作用
- 调试阶段不应默认一路把错误结果推送到末端

### 生产模板

- 每个模板保存自己的默认阶段开关状态
- 一旦模板验证通过，就按模板默认状态执行
- 如果某个模板是全流程模板，它的默认状态可以是全开

### 本次运行

- 每次运行时仍允许临时关闭某个阶段
- 也允许临时开启原本关闭的阶段

### 执行前预检

所有运行都必须先执行 `preflight`，至少检查：

- 观远凭据是否存在
- 目标页面或图表是否仍可访问
- 模板中引用的筛选器是否仍有效
- Excel 底板路径和 sheet 是否存在
- 飞书 token、sheet、range 是否配置完整
- 本次运行启用的阶段是否具备依赖条件

## 主数据模型

### 资源模型

- `folder`
- `page`
- `custom_report`

### 页面快照模型

用于持久化某次发现到的页面定义，包括：

- 页面基础信息
- `cards`
- `filterLayout`
- `tabMap`
- `dsInfos`
- 可导出图表列表

### 筛选器模型

每个筛选器至少需要保存：

- `cdId`
- `name`
- `cdType`
- `selectorType`
- `filterType`
- 数据源 `dsId`
- 字段 `fdId`
- 默认值
- 可作用的目标图表 `targetCdIds`
- 字段映射 `columnMappings`

### extract 模型

`extract` 是共享抽取单元，不直接等价于单一页面。

它应包含：

- `extract_id`
- `page_id`
- `chart_id`
- `chart_name`
- `filter_template`
- `dynamic_rules`
- 文件归档命名规则

### job 模型

`job` 表示一条业务链路。

它应包含：

- `job_id`
- 使用哪些 `extract`
- 阶段顺序
- 各阶段配置
- 默认开关
- 运行归档位置

### 运行上下文

运行时上下文用于渲染动态日期和本次覆盖配置，例如：

- `run_date`
- `timezone`
- `date_presets`
- `stage_overrides`
- `output_root`

## 配置组织

第一版推荐结构：

```text
config/
  project.json
  extracts/
    *.json
  jobs/
    *.json
  pages/
    *.json
runs/
  YYYY-MM-DD/
    <run_id>/
      manifest.json
      logs/
      downloads/
      workbook/
      publish/
```

### `project.json`

保存：

- 默认时区
- 归档目录
- 观远站点地址
- 是否启用飞书模块
- 是否启用 Excel 模块
- UI 默认选项

### `pages/*.json`

保存发现阶段抓到的页面快照，供模板编辑界面复用和差异比对。

### `extracts/*.json`

保存单个导出模板，重点是图表与筛选器组合。

### `jobs/*.json`

保存一条完整执行链路，以及各阶段的默认开关和配置。

## 本地界面设计

### 页面结构

第一版建议采用三栏布局：

1. 左侧：资源树
   - 目录
   - 页面
   - `CUSTOM_REPORT`

2. 中间：页面详情
   - tab / panel
   - 图表列表
   - 筛选器列表
   - 图表与筛选器关系

3. 右侧：模板编辑与执行区
   - 选择导出对象
   - 配置筛选器动态规则
   - 配置阶段开关
   - 配置 Excel/飞书目标
   - 执行与查看日志

### 用户核心操作

1. 刷新页面树
2. 点开页面看图表和筛选器
3. 勾选需要导出的图表
4. 把筛选器保存为动态模板
5. 选择启用哪些阶段
6. 保存为可复用模板
7. 执行单次运行
8. 查看归档与错误

## Excel 与飞书策略

### Excel

Excel 不是观远发现逻辑的一部分，而是执行层的一部分。

第一版优先支持：

- `append_rows`
- `replace_range`
- `replace_sheet_data`
- `overlay`

并保留这些增强项：

- `anchor`
- `detect_last_row_by_col`
- `clear_policy`
- `preserve_header_rows`
- `chunk_size`
- `writer_engine`

### 飞书

飞书第一阶段只支持 `Sheets`，不优先做 `Wiki`。

飞书阶段必须是可选阶段，不应强制绑定在所有任务上。

## 日志与归档

每次运行都应记录：

- 运行 ID
- 模板版本
- 启用阶段
- 实际筛选条件快照
- 观远返回的任务 ID / 文件名
- Excel 写入摘要
- 飞书写入摘要
- 失败位置

归档必须支持：

- 下载文件复查
- 参数复查
- 后续复跑
- 故障定位

## 风险与未决项

### 已确认但需工程化处理的点

- 页面树和页面详情接口的返回结构在历史记录与当前线上间存在轻微包装差异，解析器必须兼容。
- 筛选器结构要保留观远原始字段，避免过度抽象后丢失信息。
- Excel 和飞书阶段需要严格前置校验，否则副作用太大。

### 当前唯一未完全锁死的接口能力

- `DS_ELEMENTS` 类型筛选器的“在线候选值拉取接口”尚未完全定位到稳定 endpoint 名称。
- 但页面详情中已经拿到了 `dsId` 和 `fdId`，因此该能力大概率可继续打通。
- 即使第一版未完全打通，也可以先做到：
  - 读取筛选器定义
  - 保存固定值
  - 支持动态日期
  - 对枚举项先做手填或缓存值模式
