## 审计范围

本次审计基于当前工作目录完成，按用户要求：

- 不读取 `.xlsx` 文件内容
- 只记录 `.xlsx` 文件名及其与脚本配置的关系
- 重点关注脚本结构、依赖、配置、历史抓包记录和可复用部分

## 当前目录结构摘要

### 根目录中的 Excel 文件

- `绩效看板.xlsx`
- `门店详情_日维度.xlsx`
- `片区执行.xlsx`
- `全国执行.xlsx`
- `执行管理字段更新版.xlsx`
- `BD薪酬－提成.xlsx`
- `bd执行.xlsx`

这些文件共同说明：

- 当前流程是“从观远导出多个文件”
- 再写入一个或多个底板工作簿
- 并不是只下载单个报表

### 记录目录

`记录/` 中保存了：

- 页面树历史记录
- 收藏页历史记录
- 页面书签历史记录
- 导出请求负载
- 导出任务查询结果

这部分资料为当前项目提供了非常重要的接口观察依据，尤其是导出链路和筛选条件结构。

### 源码目录

`src/` 主要包含：

- `main.py`
- `api/`
- `config/`
- `process/`
- `utils/`

整体风格是“单次业务脚本 + 若干工具函数”，而不是长期可维护的应用结构。

## 当前脚本的真实工作流

从 `src/main.py` 和配置文件看，当前脚本的主流程是：

1. 读取 `.env`
2. 读取 `src/config/config.json`
3. 用固定 chart id 和固定筛选器模板发起观远导出
4. 下载多个 `.xlsx`
5. 根据映射把导出结果写入本地 Excel 底板
6. 触发 Excel 计算
7. 读取结果区域
8. 分批回写到飞书 Sheets

也就是说，它本质上已经是一个简化版流水线，只是：

- 阶段没有被显式建模
- 配置没有被抽象成通用模板
- 阶段开关没有独立存在
- 可视化选择不存在

## 旧脚本可复用部分

### 可复用 1：观远基础导出客户端

`src/api/guanbi_client.py` 已经证明以下链路可用：

- 登录
- 发起导出任务
- 查询任务状态
- 下载导出结果

这些逻辑可以作为后续新客户端的参考，但需要重写为：

- 更清晰的类型模型
- 更稳定的错误处理
- 更强的返回结构兼容性

### 可复用 2：已有筛选器样本

`src/config/config.json` 和 `记录/all_ploy.txt` 里保留了多种真实筛选样本：

- `BT` 日期区间
- `IN` 日期列表
- `STRING` 类型日期
- `DATE` 类型日期
- 日期与固定维度并存

这对后续设计动态筛选模板非常有价值。

### 可复用 3：Excel/飞书业务经验

虽然实现粗糙，但旧脚本已经沉淀了这些真实业务经验：

- 导出文件名与目标 sheet 的对应关系
- 某些底板 sheet 的粘贴起始列
- 大表飞书回写需要分批
- 部分业务表需要额外补日期列

这些经验应被迁移到模板模型中，而不是继续留在主流程代码里。

## 主要问题清单

## 问题 1：环境声明与代码依赖不一致

当前环境文件和实际代码依赖不一致。

表现：

- `requirements.txt` 包含 `python-dateutil`
- `environment.yml` 的 pip 依赖里没有 `python-dateutil`
- 代码直接 `from dateutil import parser`
- `src/utils/xlwings_utils.py` 直接 `import numpy as np`
- `environment.yml` 里也没有 `numpy`

影响：

- 项目环境无法稳定复现
- 直接导入模块就会失败

## 问题 2：入口批处理仍绑定旧路径

`src/get_bi.bat` 仍然写死旧目录和旧 Anaconda 路径。

影响：

- 当前工作目录不能直接复用
- 迁移环境后容易静默失败

## 问题 3：Excel 生命周期管理存在明显 bug

`src/utils/xlwings_utils.py` 中的 `get_actual_data_range`：

- 在 `finally` 中先 `app.quit()`
- 后面仍尝试继续访问 `wb` 和 `ws`

这是实际逻辑错误，不是单纯风格问题。

影响：

- 资源已释放后继续访问对象
- 函数在真实运行时可能出现不可预测行为

## 问题 4：飞书客户端不是通用实现

`src/api/feishu_client.py` 里存在这些问题：

- 固定 `receive_id`
- `Optional` 类型未导入
- 上传/发送逻辑仍偏向一次性脚本

影响：

- 不适合做成通用发布层
- 代码本身存在潜在导入错误

## 问题 5：主流程高度硬编码

`src/config/config.json` 写死了：

- 指定 chart id
- 指定底板文件
- 指定粘贴列
- 指定飞书表格 token

影响：

- 当前脚本只能服务少量固定业务场景
- 无法从“账号能访问哪些页面”动态扩展

## 问题 6：阶段之间强耦合

当前 `main.py` 把以下能力写在一条主链中：

- 观远导出
- Excel 写入
- Excel 读取
- 飞书上传

影响：

- 任何一个阶段出错都会影响整条链
- 无法只运行其中一段
- 无法自由插入新阶段

## 问题 7：错误边界与预检不足

当前代码缺少：

- 模板级校验
- 页面/图表存在性校验
- Excel 目标存在性校验
- 飞书目标存在性校验
- 运行前总预检

影响：

- 失败通常发生在运行中后段
- 排障成本高

## 问题 8：配置模型过于扁平

当前配置主要还是“把请求负载和写入规则堆在一个文件里”，没有形成：

- 页面快照模型
- 抽取模型
- 任务模型
- 阶段模型
- 运行上下文模型

影响：

- 无法表达复杂而可复用的模板
- 无法表达多个任务共享同一抽取结果

## 问题 9：导出结果与目标 sheet 的匹配方式脆弱

`difflib.get_close_matches(...)[0]` 依赖名称模糊匹配且没有空结果保护。

影响：

- 文件名稍有变化就可能取不到
- 直接索引 `[0]` 存在运行时异常风险

## 问题 10：历史设计稿范围已经不足

`2026-03-18-guanbi-feishu-pipeline-design.md` 主要解决的是：

- 观远到 Excel 到飞书的通用管线

但当前项目范围已经扩展为：

- 账号级页面发现
- 页面详情与筛选器建模
- 可视化选择
- 阶段化执行
- 可插入阶段
- 运行归档与复现

因此需要新的主设计稿承接完整范围。

## 当前建议的迁移策略

### 保留的东西

- 观远导出链路的业务知识
- 真实筛选样本
- Excel/飞书业务映射经验

### 重写的东西

- 主流程入口
- 配置模型
- 观远客户端结构
- Excel 执行层
- 飞书发布层
- 日志与归档模型

### 淘汰的东西

- 依赖模糊匹配的主链路逻辑
- 写死目录和写死单业务目标的批处理入口
- 把所有阶段揉进 `main.py` 的组织方式

## 对新系统的要求

新系统必须满足：

- 页面发现和执行分层
- Excel 和飞书是可选阶段
- 阶段可启用、禁用、替换、插入
- 模板可保存默认阶段开关
- 每次运行可临时覆盖
- 所有关键输入输出可归档

## 当前结论

旧脚本不是“完全推倒重来”。

更准确地说：

- 业务链路验证结果可复用
- 代码组织和配置组织必须重做
- 当前工程工作的重点是把脚本经验抽象成稳定模型和标准阶段
